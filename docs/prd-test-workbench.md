# 测评生产与验证工作台 PRD

## 1. 背景

SBTI、小鸟人格测试、红楼梦人物测试、哲学家测试等内容的流行，说明一类“低门槛作答 + 身份标签 + 可讨论结果”的测试范式具备传播潜力。它们未必依赖严肃心理测量，而更依赖快速完成、结果可解释、标签有社交表达价值、内容容易被二创和变体化。

团队希望趁高考及志愿填报相关流量窗口，探索“辅助专业选择”“天赋探索”等测评题型。团队共 4 人，异地协作；只有一人具备较强技术背景，且前端经验有限，预计大量依赖 coding agent / vibe coding。因此，当前阶段的产品目标不是直接建设最终面向公众的测评平台，而是先建设一个内部使用的测评生产与验证工作台。

工作台应帮助团队快速从想法、文档或结构化文本生成可作答的问卷，迭代题目与计分配置，收集内部成员和小样本外部测试者的作答、反馈、结果向量与 AI debug 解读，从而判断某套测评是否值得继续包装为公众传播产品。

## 2. 产品定位

本产品是一个内部工具，而不是最终消费者产品。

v1 的核心价值是：

1. 快速创建多套测评问卷。
2. 将问卷配置固化为可复盘的版本。
3. 支持内部成员和外部测试者作答。
4. 稳定生成维度结果向量。
5. 用维度解释和 AI debug 解读帮助团队判断测评质量。
6. 保存完整快照、反馈和导出数据，支撑后续人工讨论、AI 辅助分析或正式产品化。

v1 不以生成精美、可转发的公众结果页为目标。公众传播页、分享海报、动态 AI 对话测评、复杂非线性计分均后延。

## 3. 目标

### 3.1 业务目标

- 让 4 人团队能够在一个共享工作台中协作创建、修改、发布和测试多套测评。
- 让团队能在 1-2 天内从一个文档或想法生成第一版可作答问卷。
- 让每套问卷经过若干轮内部或小样本外部测试后，能基于作答结果、向量、AI 解读、反馈评分和备注判断是否值得继续包装。
- 让数据可导出，便于后续用表格、AI 或脚本进行分析。

### 3.2 产品目标

- 支持多问卷管理。
- 支持结构化文本/JSON 作为问卷事实源。
- 支持 AI 辅助把原始文本整理成问卷 schema。
- 支持 AI 辅助修改题目、选项、维度描述、开放题评分 prompt。
- 支持线性题流作答。
- 支持选择题固定向量加分。
- 支持开放题 AI 结构化评分。
- 支持结果向量、维度解释、Top/Bottom 特征、内部 AI debug 解读。
- 支持草稿、发布版本、版本对比和版本备注。
- 支持作答快照、反馈评分、备注和完整导出。

### 3.3 工程目标

- 架构克制，适合 2 核 2G 阿里云 ECS 部署。
- 避免复杂前端状态管理，优先实现可靠的文本编辑、预览、发布和作答链路。
- 所有 AI 调用必须在服务端完成，API key 不出现在前端。
- 数据库优先使用 SQLite，降低部署和运维成本。
- 保持代码和数据模型可迁移，后续可演进为正式测评产品。

## 4. 非目标

v1 明确不做：

- 不做公开注册、找回密码、复杂权限体系。
- 不做完整可视化拖拽问卷编辑器。
- 不做公众传播用精美结果页。
- 不做分享海报、社交平台自动分享。
- 不做动态 AI 追问、AI 生成下一题、对话式题流。
- 不做分支跳题。
- 不做公式、函数、脚本、条件权重等复杂计分。
- 不执行用户配置中的任意代码。
- 不承诺心理测量学严谨性、升学建议准确性或专业选择结论的可靠性。
- 不优先做移动端高度精修，但作答页应在手机上可用。

## 5. 用户角色

### 5.1 管理员/技术负责人

通常是团队中具备技术背景的人。

能力：

- 配置环境变量和 AI API。
- 设置管理访问口令。
- 管理成员。
- 创建、编辑、发布问卷。
- 查看所有数据。
- 导出数据。

### 5.2 团队成员

团队内部成员，不需要独立密码登录，但应在工作台中有成员身份。

能力：

- 进入后台工作台。
- 选择自己的成员身份。
- 创建或修改问卷。
- 发布版本并填写版本备注。
- 参与作答和反馈。
- 查看结果、历史版本和导出数据。

### 5.3 外部测试者

通过测试链接+对应口令访问某个已发布问卷版本。

能力：

- 填写姓名/昵称和备注。
- 作答问卷。
- 查看自己本次作答的结果向量、维度解释和 AI debug 解读。
- 提交反馈评分和文字备注。

限制：

- 不能访问后台。
- 不能编辑问卷。
- 不能查看版本历史。
- 不能查看他人作答。
- 不能导出数据。

## 6. 访问控制

### 6.1 后台管理访问

后台使用一个共享管理口令保护。

访问流程：

1. 用户打开后台地址。
2. 输入管理口令。
3. 通过后进入工作台。
4. 从成员列表选择当前操作者身份。

说明：

- 成员身份只用于记录“谁创建、谁修改、谁发布、谁反馈”，不承担认证安全职责。
- v1 不需要成员独立密码。
- 管理口令通过服务端环境变量配置。

### 6.2 外部测试链接

每个发布版本可生成一个随机测试链接或 token。

测试链接访问流程：

1. 测试者打开链接。
2. 进入该发布版本的作答页。
3. 填写姓名/昵称和作答前备注。
4. 完成作答。
5. 查看本次结果。
6. 提交反馈评分和文字备注。

链接规则：

- token 应随机且不可轻易枚举。
- token 对应某个问卷的某个发布版本。
- 默认启用；后台可以禁用某个测试链接。
- 每个 token 应设置作答数上限，默认 50，可在后台调整。
- 达到作答上限后，测试链接应停止接收新提交，并显示“测试名额已满”之类的友好提示。
- v1 可不做过期时间，但数据模型应预留 `disabledAt` 或类似字段。

结果链接规则：

- 结果页不得使用自增 `responseId` 作为外部可访问地址。
- 每条作答必须生成不可枚举的 `resultToken`，或直接使用 UUID/nanoid 作为 Response 主键。
- 外部结果页只接受随机 token 访问，例如 `/r/{resultToken}`。
- token 只允许查看该条作答的结果和反馈提交状态，不能枚举或访问他人作答。

## 7. 核心工作流

### 7.1 创建问卷

1. 团队成员进入后台。
2. 从成员列表选择当前操作者身份。成员系统属于 v1 基础能力，Milestone 1 至少需要一张最简 `Member` 表。
3. 点击“新建问卷”。
4. 填写问卷基础信息：
   - 标题
   - 简短说明
   - 创建者，默认使用当前选择的成员
   - 目标场景，例如高考专业选择、天赋探索、人格标签、混合型
   - 内部备注
5. 进入问卷编辑页。

### 7.2 AI 生成问卷草稿

1. 成员粘贴原始文本、题目构思、文档内容或测试说明。
2. 选择生成模式：
   - 从文档整理为 schema
   - 改写现有题目
   - 生成维度定义
   - 生成选项评分向量建议
   - 生成开放题评分 prompt
3. 服务端调用 AI。
4. AI 返回结构化问卷草稿。
5. 系统校验 schema。
6. 成员确认后写入当前草稿。

AI 生成必须经过人工确认，不应自动覆盖现有草稿。

### 7.3 编辑草稿

v1 编辑器采用“结构化文本 + 实时预览”。

编辑区：

- 显示问卷 schema 文本，建议使用 JSON 或 YAML。若工程实现更熟悉 JSON，可先只支持 JSON。
- 支持格式校验。
- 支持保存草稿。
- 支持 AI 辅助修改局部内容。

预览区：

- 展示问卷标题、说明、维度、题目、选项、开放题输入框。
- 展示每个选项对应的向量贡献。
- 展示开放题评分 prompt 摘要。
- 展示 schema 校验错误。

v1 不做拖拽式编辑器。

### 7.4 发布版本

1. 成员在草稿稳定后点击“发布版本”。
2. 系统要求填写版本备注，例如“重写第 3 题，降低艺术维度权重”。
3. 系统校验 schema。
4. 通过后固化为 `v1/v2/v3...`。
5. 发布版本生成完整不可变 JSON 快照。
6. 发布版本可生成外部测试链接。

只有发布版本可以被正式作答。

草稿可以预览，但草稿作答不进入正式数据。若实现复杂，v1 可只做页面预览，不做草稿完整作答。

### 7.5 版本对比

后台可查看某问卷的版本列表。

每个版本显示：

- 版本号
- 发布时间
- 发布者
- 版本备注
- 题目数量
- 维度数量
- 作答数量

版本对比 v1 采用简化方案：

- 支持选择两个版本。
- 展示结构化或文本 diff。
- diff 至少覆盖：
  - 标题/说明变化
  - 维度定义变化
  - 题目增删改
  - 选项文案变化
  - 选项 `deltaVector` 变化
  - 开放题评分 prompt 变化

v1 不要求精美可视化 diff。

### 7.6 作答

作答入口：

- 后台成员可选择某个发布版本作答。
- 外部测试者通过测试链接作答。

作答前：

- 填写姓名/昵称。
- 填写身份/备注，可选。
- 后台成员可关联成员身份。

作答中：

- 线性题流。
- 支持单选题。
- 支持多选题。
- 支持开放题。
- 不支持分支跳题。
- 不支持动态 AI 生成下一题。
- 前端应将未提交的作答进度保存到 `localStorage`，避免手机锁屏、刷新或误返回导致答案丢失。

提交后：

1. 提交接口立即校验并落库原始答案。
2. 系统返回 `responseId` 和初始评分状态，目标是在 2 秒内完成 HTTP 响应。
3. 系统将开放题 AI 评分和结果 debug 解读放入异步任务执行。
4. 选择题向量贡献可同步计算并立即展示为部分结果。
5. 开放题之间应并发评分；全部开放题评分完成后重算最终向量。
6. debug 解读依赖最终向量，应在开放题评分完成后执行。
7. 结果页通过轮询或轻量刷新读取评分状态，逐步点亮开放题评分、最终向量和 debug 解读。
8. 用户可在评分未完全完成时停留等待，或稍后通过结果链接/后台记录查看。
9. 用户提交反馈评分和文字备注。

提交接口必须具备幂等能力：

- 前端提交时生成 `clientSubmissionId`。
- 同一发布版本、同一测试链接/成员、同一 `clientSubmissionId` 的重复提交应返回同一个 `responseId`。
- 刷新或网络重试不能制造多条重复作答。

### 7.7 结果展示

结果页面向内部成员和外部测试者都可见，但默认展示粒度不同。

内部成员默认可见完整 debug 内容。外部测试者默认只看最终向量、维度解释、Top/Bottom 特征和 AI debug 解读，不显示每题贡献和开放题 `deltaVector` 细节，避免测试者反推出计分机制后污染二次作答或转发后的反馈。

发布版本应提供 `externalResultDetailLevel` 设置：

- `summary`：外部默认，仅展示总结果、维度解释和 debug 解读。
- `detailed`：外部也展示每题贡献和开放题评分细节，仅用于可信小范围测试。

展示内容：

- 最终向量。
- 每个维度的名称、说明、得分。
- 得分最高的若干维度。
- 得分最低的若干维度。
- 每道题的贡献明细，内部可见，外部仅在 `externalResultDetailLevel = detailed` 时可见。
- 开放题 AI 评分的 `deltaVector`、置信度和理由，内部可见，外部仅在 `externalResultDetailLevel = detailed` 时可见。
- AI debug 解读。

AI debug 解读要求：

- 语气可读，但不追求传播文案。
- 明确说明这是内部测试解释，不是确定性诊断。
- 尽量引用维度得分和关键回答。
- 帮助团队判断“这个测试是否产生了有意思、可解释、可包装的结果”。

### 7.8 反馈

作答完成后，用户可提交反馈。

反馈字段：

- 有趣程度：1-5
- 准确程度：1-5
- 愿不愿分享：1-5
- 对专业/天赋探索是否有帮助：1-5
- 整体备注

v1 可暂不做逐题评论系统。若开发成本可控，可在结果页为每道题提供一个简短批注字段，但这不是第一优先级。

数据模型应预留 `questionComments` 字段，用于未来保存按题目 ID 组织的逐题反馈。v1 可以只实现整套测试反馈 UI。

### 7.9 数据导出

后台支持导出：

- 问卷列表。
- 某问卷当前草稿。
- 某问卷所有发布版本。
- 某发布版本的作答记录。
- 作答答案。
- 每题向量贡献。
- 开放题 AI 评分结果。
- 最终结果向量。
- AI debug 解读。
- 反馈评分和备注。

导出格式：

- JSON 必做。
- CSV 可选但推荐，用于反馈评分和作答摘要。

## 8. 问卷 Schema

v1 schema 应表达以下信息。

```json
{
  "title": "专业选择探索测试",
  "description": "用于内部测试的混合型专业/天赋探索问卷",
  "scenario": "hybrid_gaokao_major_exploration",
  "dimensions": [
    {
      "id": "analytical",
      "name": "分析推理",
      "description": "偏好结构化问题、逻辑拆解和证据推理的倾向",
      "lowLabel": "直觉/体验驱动",
      "highLabel": "分析/逻辑驱动"
    }
  ],
  "questions": [
    {
      "id": "q1",
      "type": "single_choice",
      "title": "遇到一个陌生问题时，你更自然的反应是？",
      "options": [
        {
          "id": "a",
          "label": "先找规律和结构",
          "deltaVector": {
            "analytical": 2
          },
          "formula": null
        }
      ]
    },
    {
      "id": "q2",
      "type": "open_text",
      "title": "描述一次你做选择时特别纠结的经历。",
      "scoringPrompt": "请根据回答评估各维度增量，输出 JSON。",
      "scoreRange": {
        "min": -2,
        "max": 2
      }
    }
  ],
  "resultDebugPrompt": "请基于最终维度向量和关键回答生成内部 debug 解读。"
}
```

### 8.1 维度

每个维度必须有：

- `id`
- `name`
- `description`

推荐有：

- `lowLabel`
- `highLabel`
- `examples`

维度数量 v1 不做硬限制，但建议 3-10 个。太多维度会降低解释性。

### 8.2 题目类型

v1 支持：

- `single_choice`
- `multiple_choice`
- `open_text`

可后续扩展：

- 排序题
- 滑杆题
- 图片选择题
- 分支题
- 对话题

### 8.3 计分模型

v1 只支持加性向量模型。

定义：

- 问卷有 D 个维度。
- 每个作答产生一个 D 维增量。
- 最终向量为所有题目增量求和。

选择题：

- 每个选项配置固定 `deltaVector`。
- 单选题取被选选项的 `deltaVector`。
- 多选题对所有被选选项的 `deltaVector` 求和。
- 选项级别预留 `formula` 字段，用于未来受限表达式计分；v1 必须忽略该字段或要求为 `null`，不得执行其中内容。

开放题：

- 服务端将题目、回答、维度定义和评分 prompt 发给 AI。
- AI 返回结构化 `deltaVector`。
- AI 返回 `confidence` 和 `rationale`。
- 系统校验维度 id 和数值范围。

v1 不支持：

- 公式计分。
- 函数计分。
- 条件计分。
- 自定义脚本。
- 前序答案影响后续权重。
- 服务端执行用户配置代码。

未来若实现 `formula`，只能采用受限表达式求值器，表达式能力应白名单化，并禁止任意代码执行、网络访问、文件访问和数据库访问。

### 8.4 归一化

v1 可先展示原始向量。

推荐同时提供简单归一化展示：

- 每个维度显示原始分。
- 系统根据发布版本 schema 自动推导每个维度的理论最小/最大值。
- 单选题按该维度所有选项中的最小/最大贡献计算。
- 多选题按该维度负向选项贡献之和与正向选项贡献之和计算。
- 开放题按 `scoreRange` 计算；若开放题没有 `scoreRange`，则该维度范围视为不可完全推导。
- 若某维度理论范围可推导，则显示百分比或区间位置；若无法可靠计算范围，则只展示原始分。

## 9. AI 合约

### 9.1 AI 使用范围

v1 允许：

- 创作侧 AI：从文本生成 schema、改写题目、改写选项、建议维度、建议评分向量、生成开放题评分 prompt。
- 评分侧 AI：对开放题输出结构化维度增量。
- 解读侧 AI：基于最终向量和关键作答生成内部 debug 解读。

v1 不允许：

- 作答过程中动态追问。
- AI 动态生成下一题。
- AI 决定题流。
- AI 自动发布问卷。
- AI 自动覆盖草稿。

### 9.2 开放题评分输出

AI 必须输出 JSON，形如：

```json
{
  "deltaVector": {
    "analytical": 1.5,
    "creative": -0.5
  },
  "confidence": 0.72,
  "rationale": "回答体现出先拆解问题再行动的倾向，但例子中也有一定直觉成分。"
}
```

服务端必须校验：

- 是否为合法 JSON。
- `deltaVector` 只包含 schema 中定义的维度。
- 所有维度值为数字。
- 数值不超过题目定义的范围。
- `confidence` 在 0-1 之间。

校验失败时：

- 记录错误。
- 可重试一次。
- 仍失败则将该开放题标记为评分失败。
- 结果页明确显示该题未成功计分。

### 9.3 重新评分

后台必须支持单条作答重新评分。

适用场景：

- 开放题 AI 评分失败。
- 团队认为某次开放题评分明显离谱。
- 修改 `scoringPrompt` 后，需要用新 prompt 重新评估某条历史作答。

重新评分要求：

- 可对单个开放题重跑 AI 评分。
- 可对整条作答重跑所有开放题评分和 debug 解读。
- 重评后必须重算 `perQuestionScores`、`finalVector` 和 `debugInterpretation`。
- 应保留旧评分记录或至少在 `AiCallLog` 中保留重评历史，便于比较 prompt 迭代效果。
- v1 必做单条重评；批量重评可作为后续增强。

### 9.4 Debug 解读输出

AI debug 解读输入：

- 问卷标题。
- 维度定义。
- 最终向量。
- Top/Bottom 维度。
- 关键选择题作答。
- 开放题回答和评分理由。

输出：

- 3-6 段可读解释。
- 一段“可能值得包装的结果亮点”。
- 一段“不稳定/需要改题的信号”。

debug 解读不得声称：

- 这是科学诊断。
- 这是确定的升学/职业建议。
- 用户应该据此做重大人生决策。

### 9.5 AI 连接自检

后台应提供“测试 AI 连接”按钮。

行为：

- 发送一次最小 AI 请求。
- 显示是否成功。
- 显示模型名、延迟、错误信息摘要。
- 不暴露 API key。

用途：

- 部署后验证阿里云 ECS 到 AI API URL 是否可达。
- 换 key、换 base URL、换模型后快速排障。

## 12. 技术建议

### 12.1 推荐技术栈

优先选择单体全栈应用，降低部署复杂度。

推荐方案：

- Next.js 全栈应用。
- SQLite 数据库。
- Prisma 或 Drizzle ORM。
- 服务端 API route 处理 AI 调用。
- Caddy 或 Nginx 反代。
- Node 进程使用 systemd 或 pm2 管理。

备选方案：

- FastAPI + 简单前端。

若主要依赖 coding agent，Next.js 生态示例更多，生成速度可能更快；若用户更熟 Python，FastAPI 调试体验可能更直观。无论选择哪种，都应保持单体部署。

### 12.2 部署约束

目标环境：

- 阿里云 ECS。
- 2 核 CPU。
- 2GB RAM。
- 约 20GB 可用磁盘。

约束：

- 不引入 PostgreSQL/MySQL 作为 v1 必需依赖。
- 不在前端暴露 AI API key。
- AI 请求需要服务端超时和错误处理。
- 大模型调用可能较慢，作答提交不得同步等待全部 AI 调用完成；提交接口应快速返回，AI 评分异步执行，结果页轮询状态。
- 开放题 AI 评分任务应支持并发执行和失败重试。

### 12.3 环境变量

至少需要：

- `ADMIN_ACCESS_PASSWORD`
- `AI_API_BASE_URL`
- `AI_API_KEY`
- `AI_MODEL`
- `DATABASE_URL`

### 12.4 AI 任务执行

v1 可采用单进程内的轻量任务队列，但必须具备清晰的状态持久化。

最低要求：

- Response 创建后写入 `aiScoringStatus = pending`。
- 服务端异步启动评分任务。
- 任务状态、错误和中间结果持久化到数据库。
- 进程重启后，后台可以识别未完成或失败的评分记录，并允许手动重评。
- 不要求 v1 引入 Redis、BullMQ 等额外基础设施。

### 12.5 SQLite 备份

作答数据、反馈和版本记录是工作台最不可再生的资产。若 v1 使用 SQLite，上线部署必须配置最低限度备份。

最低方案：

- 使用 cron 每天执行一次 SQLite 在线备份。
- 至少保留最近 7 份备份。
- 备份目录应与应用目录分离。
- 定期手动验证备份文件可打开。

推荐命令形态：

```sh
sqlite3 app.db ".backup '/backup/app-YYYY-MM-DD.db'"
```

增强方案：

- 使用 Litestream 或同类工具将 SQLite 持续复制到 OSS/S3 兼容存储。
- 在正式收集外部测试数据前完成一次恢复演练。

## 13. 里程碑

### Milestone 1: 本地可用的问卷引擎

目标：

- 最简成员系统可用。
- 能创建问卷。
- 能编辑 schema。
- 能发布版本。
- 能通过后台作答。
- 能生成选择题加性向量。
- 能查看结果向量。

验收：

- 可手动创建成员，并在后台选择当前操作者。
- 至少一套示例问卷可完整跑通。
- 发布版本后修改草稿不影响旧版本作答。
- 作答记录通过 `versionId` 绑定不可变发布版本。
- `QuestionnaireVersion.schemaSnapshot` 发布后不可修改。

### Milestone 2: AI 与反馈闭环

目标：

- AI 从文本生成 schema 草稿。
- 开放题 AI 异步评分。
- AI debug 解读。
- 单条作答重新评分。
- AI 连接自检。
- 反馈评分和备注。
- 数据导出 JSON。

验收：

- 提交接口必须在 2 秒内返回 `responseId` 和评分状态。
- 结果页能够轮询并展示 `aiScoringStatus`。
- 多个开放题评分应并发执行，而不是在一次 HTTP 请求中串行阻塞。
- 开放题能返回合法 `deltaVector`、`confidence`、`rationale`。
- AI 输出异常时有错误提示，不导致整条作答丢失。
- 对评分失败或需要复核的作答，可从后台触发单条重评并重算最终向量。
- 后台可测试 AI 连接并展示成功/失败、延迟和错误摘要。
- 结果页能显示维度解释、Top/Bottom、每题贡献和 debug 解读。
- 导出的 JSON 包含问卷版本、答案、分数、反馈。

### Milestone 3: 协作与外测

目标：

- 版本列表和版本对比。
- 外部测试链接。
- 作答数据列表与筛选。
- CSV 导出。
- 作答进度本地恢复。

验收：

- 每个发布版本可生成随机测试链接。
- 每个测试链接可设置作答数上限，达到上限后拒收新提交。
- 外部测试者只能访问作答、结果和反馈页面。
- 外部结果页使用不可枚举的 `resultToken`，不能通过自增 ID 猜测访问他人结果。
- 外部测试者默认看不到每题贡献和开放题 `deltaVector`，除非发布版本设置为 `detailed`。
- 版本对比能看出题目、选项、维度和评分向量变化。
- 作答过程中刷新页面后可恢复本机未提交答案。

## 14. 成功指标

### 14.1 工具效率指标

- 从文本想法到第一版可作答问卷不超过 1-2 小时。
- 修改草稿并发布新版不超过 10 分钟。
- 团队能同时维护多套问卷。

### 14.2 测评筛选指标

每套问卷应能查看：

- 完成人数。
- 平均有趣程度。
- 平均准确程度。
- 平均分享意愿。
- 平均帮助程度。
- 常见反馈备注。
- 结果向量分布。

用于判断是否继续包装：

- 团队主观认为结果有解释力。
- AI debug 解读能产生可理解的亮点。
- 测试者愿意分享评分较高。
- 反馈中出现“像我”“有意思”“想发给别人”等信号。

## 15. 风险与应对

### 15.1 范围膨胀

风险：

- 想同时做编辑器、公众页、动态 AI、统计分析、分享传播，导致 v1 迟迟不能用。

应对：

- v1 坚持结构化文本编辑。
- 固定线性题流。
- 只做加性向量计分。
- 结果页只做 debug，不做公众传播包装。

### 15.2 AI 输出不稳定

风险：

- AI 返回非 JSON、维度缺失、数值越界。
- 阿里云 ECS 到 AI API URL 不可达或延迟过高。

应对：

- 服务端强校验。
- 失败重试。
- 失败时保留原始回答并显示评分失败。
- 不让 AI 自动覆盖草稿或发布版本。
- 后台提供 AI 连接自检按钮。
- 部署前必须在目标 ECS 上验证 AI API 可达性。

### 15.3 同步 AI 评分导致提交超时

风险：

- 多道开放题加 debug 解读需要多次大模型调用，若全部放在一次 HTTP 请求中串行执行，容易超过反向代理、浏览器或移动网络超时限制。
- 用户可能以为提交失败而刷新，造成重复提交或流失。

应对：

- 提交接口 2 秒内落库并返回 `responseId`。
- AI 评分异步执行。
- 结果页轮询评分状态。
- 开放题评分并发执行。
- 提交接口使用 `clientSubmissionId` 保证幂等。

### 15.4 版本污染

风险：

- 问卷修改后旧作答失去参照。

应对：

- 作答必须绑定发布版本。
- 发布版本保存完整 `schemaSnapshot`，且发布后不可变。
- Response 只引用不可变 `versionId`，不重复保存 schema。
- 只有发布版本可正式作答。

### 15.5 伪科学与误导

风险：

- 专业选择/天赋探索容易被理解为严肃建议。

应对：

- debug 解读明确标注内部测试属性。
- 外部测试结果页避免确定性升学建议。
- 文案强调探索与讨论，不做诊断或决策承诺。

### 15.6 ICP 备案与访问方式

风险：

- 外部测试链接需要测试者访问 Web 服务。
- 若在中国内地阿里云 ECS 上绑定域名提供网站服务，通常需要提前完成 ICP 备案；备案周期可能与高考/志愿填报窗口冲突。

应对：

- 短期小范围外测可先使用 IP + 端口访问，虽然不美观但能更快验证。
- 若要绑定域名，应尽早启动备案流程。
- 若备案来不及，可评估临时使用不需要内地备案的部署环境或隧道方案，但要注意访问速度、稳定性和数据安全。

## 16. 开放问题

以下问题不阻塞 v1，但需要后续决策：

- 正式公众产品是否采用混合型包装：身份标签 + 专业/天赋建议。
- 公众结果页是否需要海报、短标题、分享文案。
- 是否需要移动端深度优化。
- 是否需要分支题流。
- 是否需要动态 AI 对话测评。
- 是否需要更严肃的统计分析和题目质量评估。
- 是否需要把优秀问卷一键导出为公众站点配置。

## 17. v1 决策摘要

- 做多问卷内部工作台。
- 做轻量成员系统，但认证仍用共享管理口令。
- 做外部测试随机链接。
- 外部测试者默认只看总结果和解读，不看计分内幕。
- 做结构化文本编辑 + 预览，不做拖拽编辑器。
- 做草稿 + 发布版本。
- 只有发布版本可作答。
- 作答通过 `versionId` 绑定不可变发布版本；发布版本保存完整 schema 快照。
- 做版本对比，先用结构化/文本 diff。
- 做选择题固定向量加分。
- schema 预留选项级 `formula` 字段，但 v1 不执行。
- 做开放题 AI 结构化评分。
- AI 评分异步执行，提交接口快速返回，结果页轮询。
- 做单条作答重评和 AI 连接自检。
- 做向量、维度解释、AI debug 解读。
- 做反馈评分和备注。
- 做完整 JSON 导出，CSV 推荐。
- 不做动态 AI 题流。
- 不做公式/函数/脚本计分。
- 不做公众传播结果页。
